Codex App onboarding for tw-law-mcp

cc-crossbeam-tw

台灣建築／室內裝修審查文件助手原型。這個頁面給已安裝 Codex App 的使用者，用最短路徑把 repo 拉下來、啟用 MCP、跑完驗收，再開始做文件輔助流程。目前已完成到 Step 6：split data、source adapters、scenario tools、hardened evaluation 與 two-stage contractor flow。

首次使用流程

1

Clone repo

在本機準備專案目錄，讓 Codex App 開啟這個 repository。

git clone https://github.com/trionnemesis/cc-crossbeam-tw.git
cd cc-crossbeam-tw
2

確認 Codex MCP 設定

Codex App 會讀取專案內的 .codex/config.toml，其中的 MCP server 指向 scripts/tw_law_mcp_stdio.py。

python3 scripts/tw_law_mcp_stdio.py
3

跑全部驗收

這個指令會聚合驗證 P0 source policy、procedure/HITL、G2 fixture、metadata、jurisdiction registry、packaging strategy、scenario matrix、split data、source adapters 與 two-stage flow。

python3 scripts/run_phase_acceptance.py
4

在 Codex App 內開始問

使用 source-bound、metadata-only 的工作方式，不要把 raw drawing 或未遮罩申請文件直接貼進 prompt。

請使用 tw-law-mcp，針對新北市室內裝修圖說審核，建立 law snapshot 並說明需要人工確認的項目。

你可以直接驗證的能力

P0 source policy

驗證官方文件來源授權、更新政策、crawl policy 與 comparison coverage。

python3 scripts/run_source_policy_acceptance.py

G2 fixture baseline

驗證 12 份 synthetic de-identified cases 與 84 個 atomic correction items。

python3 scripts/run_fixture_pipeline.py

Packaging strategy

驗證 Codex/Claude Code wrappers 都指向 standalone MCP server。

python3 scripts/run_packaging_acceptance.py

Scenario matrix

驗證台灣場景 fixture 已宣告 corpus packs、tool boundary、artifact、gate 與 HITL policy；MVP 六類各至少 5 題。

python3 scripts/run_scenario_matrix_acceptance.py

Split data

驗證五個 source packs、三個 registries、兩個 fixture files 與 normalized source units。

python3 scripts/run_data_layout_acceptance.py
python3 scripts/run_source_adapter_acceptance.py

Two-stage flow

驗證 analysis → response package 骨架，不輸出合規保證、消防設計結論或材料真偽結論。

python3 scripts/run_two_stage_flow_acceptance.py

資料邊界

可以交給工具的內容

  • 已遮罩的公文文字。
  • 檔名、圖號、頁名、sheet type 等 metadata。
  • fixture 或 synthetic de-identified cases。
  • 法規 snapshot、acceptance output、HITL answers。

不要直接貼進 prompt 的內容

  • 未遮罩的設計圖、title block、地址、姓名、電話、身分證字號。
  • raw PDF、raw drawing、完整申請文件影像。
  • 任何客戶授權範圍不明的資料。
  • 可被包裝成法律意見或審查保證的結論。

使用範例：指令與期望回答

0

判斷上傳資料屬於哪個程序

適合你已經完成 local MCP 設定，並提供已遮罩公文文字、檔案 metadata 或 sheet manifest 後使用。

請使用 tw-law-mcp 先跑 run_phase_acceptance。
接著根據我提供的已遮罩文件文字與檔案 metadata，判斷目前比較接近：
圖說審核、竣工查驗、變更使用併室內裝修竣工查驗、簡易室內裝修。
請輸出 procedure_stage 信心分數、需要的人工確認問題、會用到的 corpus packs、產出的 artifacts，以及不能判定的原因。
不要輸出法律意見、合規保證、消防設計結論、材料真偽結論或審查必過承諾。

期望回答結果：Codex 應先驗證 MCP gates，再使用 resolve_procedure_stage_confidence、resolve_tw_scenario 與 two-stage flow artifacts 做程序分流；低信心、缺資料或需要專業判斷時，必須輸出 human review required，而不是硬判定。

1

先驗證工具狀態，再開始查詢

適合首次 clone 或切換 branch 後使用，確認本機工具與 acceptance gates 都可重跑。

請使用 tw-law-mcp，先列出目前可用工具，接著執行 run_phase_acceptance。回答時請用「已通過 / 需要處理 / 不能判定」三段整理結果。

期望回答結果：Codex 應列出 MCP tools，實際執行 python3 scripts/run_phase_acceptance.py，回報每個 gate 的通過狀態；若失敗，指出失敗 gate、錯誤訊息與下一步，不把驗收結果包裝成法律或審查保證。

2

建立來源可追溯的圖說審核 snapshot

適合要先確認法規來源、版本日期、資料邊界，再進入文件整理流程。

請用 build_law_snapshot 建立 jurisdiction={central:"TW", local:"ntpc"}、case_type="室內裝修"、procedure_stage="圖說審核"、as_of_date="2026-07-06" 的法規快照。回答請包含 snapshot id、來源 URL、authority rank、license 狀態，以及哪些部分需要人工確認。

期望回答結果：Codex 應回傳 snapshot metadata 與 source-bound 條目摘要，保留來源 URL、authority rank、license/update policy 與 as-of date；遇到低信心、需專業判斷或缺少使用者提供資料時，應明確標成 human review required。
